To debug bash, sometimes you need to have the debugger image run from the image
names of "bash" or "sh", because that can trigger a change of behavior, and
bash also behaves slightly differently when launched from a C program like
bash.

You can copy the bashdebug.exe image over the image in the new_gnu: directory
tree as bash.exe or sh.exe.  For the debugger to find the dsf file, it has to
also be copied as bash.dsf or sh.dsf, and the logical name DGB$IMAGE_DSF_PATH
seems to also be needed.

You can call CRTL routines like decc$perror("test") from the debugger by
doing a 'SET IMAGE DECC$SHR' before the 'call decc$perror("test")' command.
The image will be dynamically set back to the program being debugged.
Make sure that you specify valid arguments to any CRTL call, as if the call
access violates, you may not be able to recover your debug session.

Section 4: Issues found.

Bash bugs:

nojobs.c should set SA_RESTART macro to 0 if it is not defined as Bash
does this in other modules.

execute_cmd.c is referencing "job_control" variable that is not defined
when JOB_CONTROL is not defined.  This has been fixed in an official patch
to the GNU Bash code.

mkbuiltins.com is not putting "#include "stdc.h" in the builtins.com that
it creates.

VMS issue:
Job control routines present in C runtime library, but not documented
enough to use.

Found in older Bash port, a VMS added macro SAVESTRING(x) is defined as a
wrapper to the macro savestring(x).  The problem is that the savestring() macro
first does a strlen(x) and then does a strcpy(buf, x).
The VMS code is passing functions for "x".  So while the existing macro
will call "x" twice, the new VMS macro will result in "x" being called up to
three times.  Reported to the GNU_VMS maintainers.

Older bash also using PATH_MAX to allocate storage for the current working
directory.  That is only 256 characters.  Needs to be 4096 on non-VAX.

Also found a memory leak in the old VMSUTIL/restore_current_directory.
We were mallocing 256 bytes each save_current_directory for the current working
directory, and not always freeing it.

I previously fixed fixed a bug in Bash 1.14.8 in execute_cmd.c where the
existing bash code was passing a function to savestring(x).  This bug
was still present in Bash 4.2.  TODO: look for this bug in other places.
